# Weekly Report Optimizer PRD（最终 V1）
**产品定位**：面向工程师个人的 AI 周报整理工具  
**核心理念**：平台无关的「周报生成核心能力」 + 可选的「文档系统适配层」  
**V1 原则**：**先通用、再集成；先稳定、再自动化**

---

## 1. 背景与动机
工程师在日常工作中通常会通过 Daily Log 的形式记录每日工作内容（Plan / Result / Issues / Notes），并在周末整理形成周报。然而在实际过程中存在以下问题：

+ Daily Log 内容零散、粒度不一致，周五集中整理成本高
+ 周报整理高度依赖人工复制 Prompt 与多平台 AI 工具
+ AI 输出结果一次性、不可控，修改经验无法沉淀
+ Daily Log 与最终周报混杂，复盘与对外交付体验不佳

用户已形成较成熟的记录习惯（内网语雀 + Daily Log 模板），但**缺少一个稳定、可控、可复用的“周报生成中间层”**。

本工具的目标并非替代现有记录平台，而是：

**将一整周的 Daily Log，稳定转化为可直接提交的工程师周报。**

---

## 2. 产品目标
### 2.1 V1 核心目标
+ 将 Daily Log 自动整理为结构清晰、专业、可信的工程师周报
+ 将 Prompt 能力产品化，避免重复人工操作
+ 保证 AI 输出结果**可控、可二次编辑**
+ 不强绑定任何具体文档平台

### 2.2 V1 非目标（明确不做）
+ ❌ 不做多人协作、管理后台
+ ❌ 不做周报发送、邮件提醒
+ ❌ 不做复杂权限体系
+ ❌ 不在 V1 中强制集成任何第三方平台 API

---

## 3. 产品架构（V1 设计原则）
### 3.1 架构分层
```plain
┌──────────────────────────┐
│   文档系统适配层（可选）  │  ← V1 只定义，不实现
│  （如：内网语雀 Adapter） │
└──────────▲───────▲───────┘
           │       │
┌──────────┴───────┴───────┐
│   Weekly Report Core      │  ← V1 核心实现
│  - Daily Log 解析         │
│  - Prompt 管理            │
│  - AI 周报生成            │
└──────────────────────────┘
```

### 3.2 设计说明
+ **Core 层平台无关**：只处理文本输入与周报输出
+ **Adapter 层可插拔**：负责“从哪来 / 到哪去”，不参与业务逻辑
+ V1 仅实现 Core 层，Adapter 作为后续扩展方向

---

## 4. 使用对象与使用场景
### 4.1 使用对象
+ 有固定 Daily Log 记录习惯的工程师
+ 希望提升周报整理效率与质量的个人用户

### 4.2 V1 使用流程
1. 周一至周五：用户在内网语雀中持续记录 Daily Log
2. 周五：用户复制整周 Daily Log 内容
3. 将内容粘贴至工具中生成候选周报
4. 人工微调后，作为最终周报提交

---

## 5. 输入规范（V1）
### 5.1 Daily Log 模板（保持用户现有习惯）
```plain
12-15 | 周一
Plan

Result

Issues

Notes

12-16 | 周二
Plan

Result

Issues

Notes
...
```

### 5.2 输入约束
+ 支持 Markdown / 纯文本
+ 按日期分段
+ Plan / Result / Issues / Notes 为推荐结构，允许部分缺失

---

## 6. 功能需求（V1）
### 6.1 Daily Log 解析
+ 按日期拆分整周内容
+ 识别 Plan / Result / Issues / Notes 四类信息
+ Notes 在 AI 整理阶段自动降权

---

### 6.2 Prompt 管理（核心能力）
+ 内置稳定 Prompt 模板
+ Prompt 规则包括：
    - 内容 100% 来源于 Daily Log
    - 不虚构、不扩大、不遗漏
    - 严格遵守周报模板结构
+ 用户无需手动复制 Prompt

---

### 6.3 AI 周报生成
+ 基于解析后的 Daily Log 调用大模型
+ 输出一份「候选周报」
+ 输出内容必须严格遵守以下模块结构：
    - 本周工作总结
    - 本周输出成果（Deliverables）
    - 问题 & 风险（Issues & Risks）
    - 下周工作计划

---

### 6.4 输出与编辑
+ 输出格式为 Markdown
+ 支持用户手动编辑
+ 支持一键复制最终周报内容

---

## 7. 输出规范（强约束）
```plain
【本周工作总结】
- xxx：
  - xxx
  - xxx

【本周输出成果（Deliverables）】
- ✓ xxx
- ✓ xxx

【问题 & 风险（Issues & Risks）】
- 问题 1：xxx  
  - 影响：xxx  
  - 需要：xxx（如有）  
- 风险 2：xxx  
  - 缓解方案：xxx  

【下周工作计划】
- 计划 1：xxx
- 计划 2：xxx
- 计划 3：xxx
```

---

## 8. 非功能需求
### 8.1 可控性
+ 不输出任何 AI 元信息或说明性文字
+ 所有内容以“候选稿”形式呈现

### 8.2 易用性
+ 操作路径 ≤ 2 步：粘贴 → 生成
+ 面向工程师用户，界面保持简洁

---

## 9. 风险与限制
+ AI 输出质量依赖 Daily Log 质量
+ Notes 过多可能干扰整理结果
+ Prompt 规则需通过真实使用持续调优

---

## 10. V1 成功标准
+ 连续 2 周使用该工具生成周报
+ 周报生成后仅需少量人工调整即可提交
+ 明显减少 Prompt 复制与平台切换成本

---

## 11. 后续演进方向（不在 V1 范围内）
+ 文档系统适配器（如内网语雀读写）
+ 多周上下文感知与连续事项识别
+ Prompt 规则配置化
+ 与「经验沉淀 / 技术方案生成」模块共享核心能力
